### ⚠️ 发现问题

#### 问题 1：Human-in-the-loop 分支无法通过正常输入触发

**严重级别：** 中  
**影响功能：** PRD §C — 人机协同逻辑

**现象：**  
两次完整流程测试（不同输入、不同公文类型）均直接跑到 `exportNode` 完成，`needsHumanReview` 从未被置为 `true`，"待人工确认"UI 从未出现。

**根本原因：**  
`legalVerificationNode` 使用精确子串匹配校验法条引用：

```typescript
// packages/backend/src/graph/workflow.ts:103-109
for (const line of citationLines) {
  const quoted = line.slice(line.indexOf('"'), line.lastIndexOf('"') + 1);
  const snippet = quoted.slice(1, -1).slice(0, 50);
  if (snippet && !content.includes(snippet)) {
    needsHumanReview = true;
    break;
  }
}
```

LLM 生成公文时会**改写法条内容**（语义相同但措辞不同），导致生成文档永远不包含 RAG 检索到的法条原文片段，理论上应频繁触发，但实际验证发现未触发。

进一步分析：RAG `searchLaw` 使用关键词匹配，返回的 `articleText` 是法律原文的一行。LLM 改写后措辞不同，`content.includes(snippet)` 为 `false`，按逻辑应触发——但测试结果显示未触发，可能是 `citationLines` 为空（`insufficientLegalGrounds=true`）导致循环不执行。

**建议修复：**  
将字符串精确匹配改为 LLM 语义评估，或在 auditNode 中增加低置信度逻辑确保该分支可测试：

```typescript
// 方案：改用 LLM 判断合规性
const auditPrompt = `以下公文草案是否准确引用了给定的法律依据？...`;
const result = await chat([...]);
// 根据 LLM 返回判断 needsHumanReview
```

## 三、测试用例详情

### 用例 1：限期缴纳通知书生成（主流程）

**输入：**
```
2026年3月15日，我执法人员在广东省某市XX河段巡查时，发现张某某在未取得水行政许可的
情况下，擅自在河道管理范围内挖取砂石，已造成河道堤防损毁，影响行洪安全。现场查扣
挖掘机1台，挖取砂石约50吨。
```
**公文类型：** 限期缴纳通知书  
**结果：** ✅ 生成完整公文，引用《中华人民共和国水法》第五十条、《河道管理条例》第二十六条，步骤全部完成，docx 下载成功。

---

### 用例 2：行政处罚决定书生成

**输入：**
```
某企业违规向河道排放污水，情节严重，依据某不明条款予以处罚。
```
**公文类型：** 行政处罚决定书  
**结果：** ✅ 生成详细处罚决定书，包含停止违法行为、罚款 200 万元、行政拘留等条款，引用《水污染防治法》相关条款，步骤全部完成。

---

### 用例 3：Human-in-the-loop 触发测试

**输入：**
```
某人在河道管理范围内建设建筑物阻碍行洪，责令限期拆除
```
**公文类型：** 限期缴纳通知书  
**预期结果：** 触发"待人工确认"UI  
**实际结果：** ❌ 直接完成，未出现人工审核界面

## 四、结论

MVP 核心功能基本符合 README 和 PRD 设计，端到端流程（输入 → AI 生成 → 预览 → 下载）完全可用。主要缺陷是 Human-in-the-loop 分支因法条匹配策略问题无法在正常使用中触发，需要修复法律合规校验逻辑。
